# 手動ログ管理 改善設計書

## 背景と目的

タイマーの押し忘れが多く、作業後に遡って記録したいケースが頻発している。
現状はタイマー起動を前提とした設計のため、以下の課題がある。

### 現状の課題

| 課題 | 現状 |
|---|---|
| 手動でのログ追加ができない | タイマーを使わずにログを作成する手段がない |
| ログの全体像が把握しにくい | タスクランキング → ドロワーの2段階でしか閲覧できない（タスク軸のみ） |
| 時間軸でのギャップが見えない | 「14:00〜16:00の間に記録がない」ことに気づけない |
| 操作フィードバックがない | 保存・削除の成功・失敗が視覚的に確認できない |
| 編集項目が限定的 | 開始・終了時刻とアクティビティのみ編集可能。時間を「1時間30分」のように直接入力できない |

---

## 設計方針

1. **ダッシュボードを手動管理の主戦場にする** — 追加・編集・削除はすべてダッシュボードで完結
2. **時系列ビューを新設する** — 時間軸でギャップを可視化し、どこに追加すべきかを直感的に把握できるようにする
3. **既存のタスク軸ビューは残す** — タスクランキング + ドロワーによる現状の操作フローは維持
4. **タスクメタ取得はキャッシュファースト・フォールバック手入力** — Content Script 経由でしかJira APIを叩けないため、キャッシュにないタスクはタイトルを手入力させる

---

## 機能仕様

### 1. 手動ログ追加

#### トリガー

- ダッシュボード：時系列ビュー内の「＋ ログを追加」ボタン
- ダッシュボード：ヘッダー付近のグローバル「＋」ボタン（時系列ビュー未表示時も操作可能）

#### 入力フォーム（モーダルまたはドロワー）

| フィールド | 入力方式 | 必須 | 備考 |
|---|---|---|---|
| タスクキー | テキスト入力（例: `PROJ-123`） | ✓ | 入力確定後にキャッシュ検索してタイトルを自動補完 |
| タスクタイトル | テキスト入力 | ✓ | キャッシュヒット時は自動補完。未ヒット時は手入力 |
| アクティビティ | セレクト | ✓ | 既存の5種類 |
| 日付 | 日付ピッカー | ✓ | デフォルト：今日 |
| 開始時刻 | 時刻ピッカー（HH:MM） | ✓ | |
| 終了時刻または所要時間 | 時刻ピッカー **または** `1h30m` 形式のテキスト入力 | ✓ | 片方を入力すると他方を自動計算 |

#### タスクキー入力時のメタ解決フロー

```
ユーザーがタスクキーを入力・確定
  └─ metaCache を検索
      ├─ ヒット → taskTitle / projectKey を自動補完（グレーのプレースホルダーで表示）
      └─ ミス  → タイトル欄を空にしてフォーカス移動（手入力を促す）
                 ※ projectKey はタスクキーのプレフィックスから自動導出（例: PROJ-123 → PROJ）
```

#### バリデーション

- `startedAt < stoppedAt`（必須）
- `stoppedAt` が未来日時でないこと（必須）
- 同日に重複するログがある場合は警告表示（ブロックはしない）

#### 保存後

- 成功トースト表示（3秒）
- 時系列ビューが表示中であれば追加したログにスクロール

---

### 2. 時系列ログビュー（新規）

ダッシュボードにタスク軸ビュー（現状のランキングテーブル）と並ぶ、**時間軸ビュー**タブを追加する。

#### レイアウト

```
[ タスク軸 | 時間軸 ]   ← タブ切り替え

[ ＋ ログを追加 ]

2026-03-29（土）
  10:00–11:30  PROJ-123  実装          1h 30m  [編集] [削除]
  11:30–12:00  PROJ-456  コードレビュー  30m    [編集] [削除]
  ▲ ギャップ: 12:00–14:00（2時間）← 強調表示
  14:00–15:00  PROJ-123  設計          1h      [編集] [削除]

2026-03-28（金）
  ...
```

#### ギャップ表示のルール

- 同日内で連続するログの間に **30分以上** の空白がある場合に「ギャップ: XX:XX–XX:XX（X時間）」をグレーの区切り行で表示
- ギャップ行クリック → 手動追加フォームをその時間帯で開く（開始・終了時刻を自動セット）

#### ソートとグループ化

- 日付降順（新しい日が上）
- 日付内では開始時刻昇順

---

### 3. ログ編集の改善

#### 編集フォームの変更点

現状の `LogEditDrawer` を以下のように改善する。

| 項目 | 現状 | 改善後 |
|---|---|---|
| 終了時刻の入力 | datetime-local | 時刻のみ（日付は開始時刻の日付を使用） |
| 所要時間の直接入力 | なし | 追加。`1h30m` / `90m` / `1:30` 形式を受け付ける |
| バリデーションエラー | サイレント（何も起きない） | エラーメッセージをインライン表示 |
| 保存フィードバック | なし | 成功トースト / エラートースト |

#### 所要時間入力と時刻入力の連動

```
開始時刻: 10:00
終了時刻: [11:30]  ←→  所要時間: [1h 30m]
          （片方を変更すると他方を自動更新）
```

---

### 4. ログ削除の改善

#### 削除後の Undo

- 削除実行直後に「削除しました　[元に戻す]」トーストを5秒表示
- 5秒以内に「元に戻す」を押すと復元
- 実際の `chrome.storage.local` への削除書き込みは5秒後に実行する（遅延削除）

#### 現状の2ステップ確認

削除確認のUIは現状のまま維持する（2度押し要求）。

## 決定事項

1. **タスクキーがキャッシュにない場合** — 手入力を許容する。`projectKey` はタスクキープレフィックスから自動導出するため集計への影響なし。
2. **ポップアップでの追加** — ポップアップにも「手動追加」ボタンを置き、クリック時に `chrome.runtime.openOptionsPage()` でダッシュボードを開く。ダッシュボード側は URL パラメータ（例: `?action=add`）を検知して追加モーダルを自動オープンする。
3. **日またぎのログ** — 許容しない。`stoppedAt` の日付が `startedAt` の日付と異なる場合はバリデーションエラーとする。
4. **アクティブタイマー中の手動追加** — 時刻重複は警告のみ（ブロックしない）。
